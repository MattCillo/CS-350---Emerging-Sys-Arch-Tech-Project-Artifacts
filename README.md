# CS-350 Emerging Sys Arch & Tech Thermostat Project Artifacts

This project shows the work completed in CS 350 and highlights my experience with emerging systems architectures and technologies. The main goal of the project was to build a working thermostat system that could read temperature data, respond to user input, control hardware components, and communicate system information in a clear way. Through this project, I was able to combine software and hardware in a way that felt practical and closer to how real embedded systems work.

The thermostat was designed to solve a simple but important problem. A user needs a system that can monitor room temperature, switch between operating modes, adjust a target temperature, and give feedback about what the system is doing. In this project, that was done by using a temperature sensor, buttons, LEDs, an LCD display, and UART communication. Together, these parts allowed the thermostat to act like a small but complete embedded system instead of just a piece of code running by itself.

One of the main artifacts in this repository is the thermostat Python code. I chose this file because it shows my ability to write software that directly controls hardware components and manages the logic that connects them. The code handles reading the temperature sensor, changing between OFF, HEAT, and COOL modes, updating the set point with button input, controlling the red and blue LEDs based on system behavior, updating the LCD display, and sending status information through UART. I think this artifact best represents the hands-on side of my work because it shows how I turned project requirements into a functional system.

The second artifact is the state machine diagram. I included this because it represents the planning and design side of the project. Before a system like this can work well, it helps to break it into clear states and define how it should move from one state to another. The diagram shows those transitions and makes the thermostat easier to understand, explain, and maintain. I think this artifact reflects my ability to stay organized and think through how the system should behave before and during development.

One part of this project that I did particularly well was connecting the code to the hardware behavior in a structured way. Instead of treating each part separately, I worked to make the buttons, display, LEDs, sensor, and communication all support the same overall thermostat logic. I also think I did well in documenting the project through the diagram, since that helped explain the system beyond just the code itself. That balance between implementation and planning is something I want to keep improving as I move into future courses and projects.

At the same time, there are areas where I could improve. Like a lot of embedded work, this project showed me that testing and debugging can take more time than expected, especially when hardware and software both have to work together. Looking back, I could improve by being even more careful with final submission details and by making sure every required deliverable is included the first time. I also think I can continue building stronger documentation habits, especially when it comes to citing technical sources and describing design decisions more completely.

This project also helped me see which tools and resources are most helpful to keep using in the future. Course guides, technical documentation, hardware pin references, and careful step-by-step testing all made a big difference. I also learned that it helps to have a support process, not just support resources. By that, I mean going one part at a time, checking output carefully, and comparing the results back to the requirements instead of guessing. That approach is something I will carry forward into other technical work.

Several skills from this project will transfer well to other classes and future projects. One is the ability to connect software logic to physical hardware behavior. Another is designing a system around states, inputs, and outputs so that the program stays organized instead of turning into a long list of unrelated actions. I also strengthened my debugging skills, because this project required patience, testing, and attention to detail. Those skills are useful in almost any area of computer science, even when the project is not focused on hardware.

I tried to make this project maintainable, readable, and adaptable by keeping the logic organized and by separating key behaviors into clear parts. The state machine design helped make the overall flow easier to follow, and the code structure made it easier to understand how the thermostat modes related to the LEDs, display, and temperature readings. That matters because maintainable code is easier to revisit later, easier to explain to someone else, and easier to improve when new features are needed. In the end, this project represents both what I built and what I learned about designing systems that are clear, practical, and easier to support over time.
